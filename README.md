### Minimal glossary

| Term | Meaning in the project |
|---|---|
| **VED** | Vehicle Energy Dataset: OBD-II + GPS telemetry of 299 vehicles in Ann Arbor (MI). |
| **MAF** | Mass Air Flow (g/s): air taken in by the engine; **a consumption proxy for ICE only**. |
| **`maf_per_km`** | The target of the consumption model: MAF integrated over the segment / km traveled = "energy cost per km" of the segment. |
| **ICE / HEV / PHEV** | Combustion / hybrid / plug-in hybrid. The consumption model uses ICE only. |
| **Map-only** | Constraint on the features: only information known **in advance** to an ACC (map, expected speed profile); RPM/load excluded. |
| **Segment** | A ~250 m stretch of a trip: the unit on which consumption is predicted. |
| **Look-ahead / `next_*`** | Features of the **next segment** (future road/speed), built with `shift(-1)`. The "predictive" heart of the ACC. |
| **`stop_fraction`** | Fraction of near-stationary time (speed<2) in the segment: the model's most important feature (stop-and-go). |
| **Trip / `VehId`** | A vehicle's trip / the vehicle's (pseudonymous) identifier. `Datetime` is constant per trip; intra-trip time is `Timestampms`. |
| **Spatial cell** | ~11×8 m bin (lat/lon to 4 decimals) over which passages are aggregated for the segment clustering (kept if ≥50 passages). |
| **Slope / `dz_net`** | Point slope / net elevation change of the segment. In Ann Arbor: a weak signal due to data limitations (elevation quantized ~111 m). |
| **Reframe** | The project's turning point (D1): from "orographic ACC" to "energy + context from the speed profile". |
| **Leakage** | Information from the target (or from the future) that leaks into the features or the validation → inflated metrics and a model unusable in the real use case. |

## Project explanation

### The problem
A classic ACC regulates speed by reacting to what it sees *now* (radar, vehicle ahead).
A **predictive** ACC could do better if it knew **what type of road it is about to face**
and **how much, in energy, a given speed profile costs** on that segment. The ML problem is
therefore twofold:

1. **Context classification** (unsupervised): given the telemetry history over a road
   segment, recognize the *type* of the segment (intersection, mixed urban, free-flowing, highway)
   and the *style* of the driver, without hand-provided labels, because they do not exist in the data.
2. **Consumption estimation** (supervised): predict the consumption per km for the **upcoming
   segment**, using only information **known in advance** (map + expected speed profile), so that the
   prediction is usable to *plan*, not just to *explain after the fact*.

### The objective
Show that the road context is recognizable from geometry alone,
  and that consumption is predictable *in advance* with good fidelity.

## Data

### The chosen dataset (and why that one)
**Vehicle Energy Dataset (VED)** — University of Michigan, public (official repo:
`github.com/gsoh/VED`): ~18.26M raw rows of real OBD-II telemetry (54 parquet files),
299 vehicles (ICE/HEV/PHEV) in Ann Arbor (MI), with GPS. After cleaning and enrichment: ~17.9M rows.

Why this data (and not other — see D2):
- **GPS + telemetry together**: without position you cannot build either the road context or
  the anticipation "what is in the next segment". An OBD-II dataset without GPS, evaluated as
  an alternative, was discarded precisely for this.
- **Observable consumption**: MAF (Mass Air Flow) is a physical proxy of consumption for
  combustion engines (fuel is metered on the intake air).
- **Heterogeneous fleet**: more vehicles and more drivers → the *styles* can be studied.

### Enrichment data
- **Elevation** via the Open-Meteo API (free, no key, SRTM base ~30 m), with a resumable
  local cache → from which **slope** via Haversine and **acceleration** on real `dt`.
  This enrichment mainly served to **measure how much the terrain is worth** — and the answer
  (almost nothing, in Ann Arbor) redesigned the project.

### What shape the "content" to classify has
- **Road segments**: spatial cells ~11×8 m (lat/lon to 4 decimals) aggregated over all passages,
  kept only if ≥50 passages (stable statistics). Features: **kinematics + geometry only**
  (speed, accelerations, stops, slope) — consumption is kept *out* of the clustering and used
  only as a descriptive column, so as not to make the result circular (D9).
- **Drivers**: profile per `VehId` (speed, accelerations, stops), deliberately
  **powertrain-agnostic** so that the style is comparable across ICE, HEV and PHEV.
- **Trip segments ~250 m** for consumption: short enough to be homogeneous, long enough
  to have stable statistics and stay above the elevation noise.

### Handling of missing and incorrectly formatted data
- **Out-of-physical-range outliers** (NB1): filters on Load 0–200%, RPM 0–8000, Speed 0–200 km/h,
  MAF 0–300 g/s → impossible sensor readings are discarded, not legitimate tails of the
  distributions.
- **"Valid motion" filter** (NB1): stationary/non-informative rows excluded before enrichment.
- **Missing elevation**: the NB1 elevation cell checks the completeness of the cache and **raises
  an exception** if uncovered points remain (a handled sad path, not ignored): the merge does not start
  with an incomplete cache.
- **Spatial cells with few passages** (NB3): `MIN_PASSAGES=50` filter → we do not classify what
  there is not enough statistical evidence for.

## ML lifecycle

### Data collection
- Consolidation of the 54 VED parquet files, EDA, outlier filters on physical ranges and the "valid
  motion" filter (details in the "Data" section).
- Elevation enrichment: dedup of GPS points to 3 decimals (~111 m) → ~8.7k unique points to
  query, resumable cache (the API is rate-limited). The 30 m grid
  (native SRTM resolution) was explored and **cancelled**: ~2h of calls for a marginal gain on a flat
  city (D10 — an example of a tracked cost/benefit decision).
- Single output: `outputs/ved_enriched.parquet`, the input to everything else.

### Training
- **Consumption**: XGBoost (default + tuned with Optuna), target `maf_per_km` per segment. The decision
  that defines the model is **map-only**: only features known in advance to an ACC are used
  (expected kinematics, geometry, look-ahead on the next segment, context), **excluding**
  RPM/engine load which would be quasi-circular with the target (MAF ≈ RPM × load) (D3).
- **Context/styles**: StandardScaler

### Validation
As seen in class, in ML systems the behavior is not deterministic and depends on the **data**,
not just on the code logic: testing adapts on three levels.
- **Data & component** (transformation tests): sanity checks in NB1 (physical ranges, sample
  trip, post-enrichment statistics) and verification that the elevation cache is complete before the
  merge (the cell *raises* if points are missing: a handled sad path, not ignored).
- **Anti-leakage**:
  - **temporal split per trip** (train on the oldest trips, test on the most recent ones): simulates the
    real use "predict the future";
  - **GroupKFold per vehicle** in the tuning: no vehicle is in both train and validation → the
    model cannot "recognize the vehicle" instead of learning the road (D12);
  - the anticipation features `next_*` are built with `shift(-1)` per (vehicle, trip) and
    carry only the **future of the road**, never the future MAF.

### Monitoring
- **Prediction quality**: continuous comparison predicted vs observed (the real `maf_per_km` is
  computable on board after the fact) → rolling MAE/R² per geographic area and per time band.
- **Data drift**: distributions of `speed_mean`, `stop_fraction`, OAT per zone vs the training
  ones; an ACC taken to a hilly city **must** trigger a drift alarm on the
  terrain (we already know the model is blind there).
- **Context health**: fraction of traversed segments that have no label (out of coverage or
  <50 passages) — if it grows, the context layer is aging.


## MLOps

### When retraining would be needed (Continuous Training)
1. **New geographic area**: everything is learned on flat Ann Arbor; on a territory
   with real orography the terrain weight must be re-measured — it could go from ~0.06 to dominant.
2. **Seasonal/fleet drift**: OAT, tires, fleet composition change (more hybrids
   → the ICE population on which the model is valid shrinks).
3. **Change of the data source**: new ECU/OBD sampling frequency, new elevation
   source (e.g. switch to a 30 m grid: already explored, D10) → feature recompute + retraining.

**Assumptions (explicit and verified where possible):**
- MAF ∝ consumption **for ICE only** (verified on the schema: no battery signal in VED).
- The historical passages over a segment describe the segment (holds only with ≥50 passages).
- `Datetime` is constant per trip; intra-trip time is `Timestampms` (verified on the data).

**Limitations (of the data, not of the method):**
- Almost flat terrain + elevation quantized to ~111 m → slope null in 92% of rows, correlation
  with MAF ~0.004: the role of orography **is not measurable here**. It is the limitation that caused the
  reframe (D1) and it is the first thing to say to whoever evaluates the project.
- Hybrids out of the consumption model (SoC is missing); PHEVs underrepresented (12 vehicles).
- K-Means clustering assumes clusters in the chosen features; the naming is interpretive.

**Difficulties encountered** (the main ones, all tracked):
1. The **orographic signal was not there** → project reframe (D1), the biggest and most
   formative difficulty.
2. **Rate-limit of the elevation API** (HTTP 429) → resumable local cache that downloads only the missing
   points.
3. **Hybrids not modelable** without SoC → motivated ICE-only scope (D4).
4. **Environment fragility**: scikit-learn version mismatch on the `.joblib` artifacts and a
   broken `.venv` to recreate → the rule "same environment to save/load" and a disposable venv.
5. **Code↔documents divergence** during fast refactoring moments → STATE.md discipline and
   coherence checks (it already happened, and it already happened to have to fix it).

### What must be clear to whoever joins the team
In reading order: **`STATE.md`** (where we are) → this README + `DECISION_LOG.md` (why we are
here) → `PROJECT_REPORT.md` (the full story) → `CLAUDE.md` (traps not to reintroduce)
→ `discussions.md` sampled. The five takeaways:

1. The project is **not** "the initial idea that succeeded": it is an idea **falsified by the data and reframed**
   — and this is its methodological value.
2. The decision that defines everything is **map-only**: never features not known in advance to an ACC.
3. Consumption is modeled **for ICE only**, for a physical reason (MAF), not out of laziness.
4. The metrics count less than the **explanation**: feature importance and cluster profiles are
   the proof, R² and silhouette the garnish.
5. If something seems arbitrary (250 m, K=4, ~111 m grid), **it is not**: there is an entry in the
   decision log with the discarded alternative and the reason.
