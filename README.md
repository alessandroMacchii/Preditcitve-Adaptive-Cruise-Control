## Project explanation

### The problem
A classic ACC regulates speed by reacting to what it sees *now* (radar, vehicle ahead).
A **predictive** ACC could do better if it knew **what type of road it is about to face**
and **how much, in energy, a given speed profile costs** on that segment. The ML problem is
therefore twofold:

1. **Context classification** (unsupervised): given the telemetry history over a road
   segment, recognize the *type* of the segment (intersection, mixed urban, free-flowing, highway)
   and the *style* of the driver — without hand-provided labels, because they do not exist in the data.
2. **Consumption estimation** (supervised): predict the consumption per km for the **upcoming
   segment**, using only information **known in advance** (map + expected speed profile), so that the
   prediction is usable to *plan*, not just to *explain after the fact*.

### The objectives (and the non-objectives)
- **Objective:** show that the road context is recognizable from kinematics+geometry alone,
  and that consumption is predictable *in advance* with good fidelity (achieved: R² ~0.76 map-only).
- **Objective:** understand **which signal actually matters**. Here is the most important lesson of the
  project: the starting hypothesis (**orography** as the key signal) was **falsified by the
  data** and the project was reframed around the **speed/traffic profile** → see
  [`DECISION_LOG.md` D1](DECISION_LOG.md). In course terms: the project was born as an **MVP of
  an idea** ("is the orographic ACC worth it?") and followed the cycle *Test → Learn → Iterate*: the
  answer of the first cycle was "no, on this data" — and the value collected was precisely
  that validated information, obtained with minimal effort.
- **Non-objective:** a production-ready system. Deploy and monitoring are treated at a conceptual
  level (sections "ML lifecycle" and "MLOps"); the real scope is an exam project with
  reproducible notebooks.
- **Non-objective:** covering hybrid/electric vehicles in the consumption model (see "Data" and D4).

### Success criterion
Not just the metric (R², silhouette), but **defensibility**: every choice must withstand the question
"why this way and not otherwise?", with verified numbers. That is why the decision log exists.

### The work process (development model)
The project was developed with an **iterative/agile** process, not waterfall — and not as a fad:
the requirements *were not knowable in advance*, because they depended on what was actually in the data.
Each work session is a small cycle: hypothesis → exploration → documented decision →
state update (`STATE.md`). The D1 reframe is the textbook example of why waterfall
would have failed here: a rigid plan around the orographic idea would have produced, with great
precision, the wrong project.

In practice each session follows the same ritual: it starts from `project context/STATE.md` (the **live
state**: what is executed, what is missing), work happens, and at the end of the session STATE.md is updated
and — if there is a decision — the decision log; the conceptual discussions become numbered entries in
`project context/discussions.md`. **Roles**: Alex is owner/decider (≈ Product Owner: scope,
priorities, notebook execution, commits); the AI pairing sessions cover preparation of
code/analysis and maintenance of the documentation. In the decision log, scope choices carry the name
of who made them, so a technical choice is distinguished from an editorial one.

## Data

### The chosen dataset (and why that one)
**Vehicle Energy Dataset (VED)** — University of Michigan, public (official repo:
`github.com/gsoh/VED`): ~18.26M raw rows of real OBD-II telemetry (54 parquet files),
299 vehicles (ICE/HEV/PHEV) in Ann Arbor (MI), with GPS. After cleaning and enrichment: ~17.9M rows.

Why this data (and not other — see D2):
- **GPS + telemetry together**: without position you cannot build either the road context or
  the anticipation "what is in the next segment". An OBD-II dataset without GPS, evaluated as
  an alternative, was discarded precisely for this.
- **Observable consumption**: MAF (Mass Air Flow) is a physical proxy of consumption for
  combustion engines (fuel is metered on the intake air).
- **Heterogeneous fleet**: more vehicles and more drivers → the *styles* can be studied.

### Enrichment data
- **Elevation** via the Open-Meteo API (free, no key, SRTM base ~30 m), with a resumable
  local cache → from which **slope** via Haversine and **acceleration** on real `dt`.
  This enrichment mainly served to **measure how much the terrain is worth** — and the answer
  (almost nothing, in Ann Arbor) redesigned the project.

### What shape the "content" to classify has
- **Road segments**: spatial cells ~11×8 m (lat/lon to 4 decimals) aggregated over all passages,
  kept only if ≥50 passages (stable statistics). Features: **kinematics + geometry only**
  (speed, accelerations, stops, slope) — consumption is kept *out* of the clustering and used
  only as a descriptive column, so as not to make the result circular (D9).
- **Drivers**: kinematic profile per `VehId` (speed, accelerations, stops), deliberately
  **powertrain-agnostic** so that the style is comparable across ICE, HEV and PHEV.
- **Trip segments ~250 m** for consumption: short enough to be homogeneous, long enough
  to have stable statistics and stay above the SRTM elevation noise (D5).

### Handling of missing and incorrectly formatted data
- **Out-of-physical-range outliers** (NB1): filters on Load 0–200%, RPM 0–8000, Speed 0–200 km/h,
  MAF 0–300 g/s → impossible sensor readings are discarded, not legitimate tails of the
  distributions.
- **"Valid motion" filter** (NB1): stationary/non-informative rows excluded before enrichment.
- **Time-format quirk**: `Datetime` is **constant per trip** (it is not a per-row
  timestamp!); intra-trip time is `Timestampms`. Whoever worked on it documented it (CLAUDE.md,
  VED_DATA_ANALYSIS.md) because using it "as is" would produce wrong time derivatives.
- **First row of each trip**: `slope`/`accel` are **fictitious 0** (not NaN) due to an `np.where`
  on NaN → ~26k rows with artifact values. Low impact, **known and documented**; in NB2 the
  `dist_m`/`dt_ms` NaNs of those rows are zeroed anyway before segmentation.
- **Missing elevation**: the NB1 elevation cell checks the completeness of the cache and **raises
  an exception** if uncovered points remain (a handled sad path, not ignored): the merge does not start
  with an incomplete cache.
- **Residual NaN at segment level** (NB2): `dropna` on features+target → segments without
  look-ahead (the last one of each trip has no `next_*`) leave the training by construction.
- **Spatial cells with few passages** (NB3): `MIN_PASSAGES=50` filter → we do not classify what
  there is not enough statistical evidence for.

### Ambiguous and delicate cases

| Case | Why it is delicate | How we treat it |
|---|---|---|
| **Hybrid vehicles (HEV/PHEV)** | In electric mode MAF=0 but the vehicle consumes: the proxy "MAF=consumption" **lies**. VED has no battery state (SoC) → the electric fraction is an unobserved latent variable. | Consumption model **ICE only** (D4). Extending it to hybrids without SoC = leakage or bias; documented as a future development (*hurdle* model with SoC). |
| **PHEV: only 12 vehicles out of 299** | Any conclusion about PHEVs rests on a tiny sample. | Conclusions marked as *indicative*; no decision founded on PHEVs alone. |
| **Low-traffic segments** | With <50 passages the per-cell means are noise: you would classify the case, not the road. | `MIN_PASSAGES=50` filter. Cost: little-used roads are not classified (declared limitation). |
| **"Mixed" segments** | A segment that is too long mixes arterial + traffic light → an ambiguous label by definition. | ~250 m segments + sensitivity check (150/250/500 m): the conclusions do not change (D5). |
| **Cluster naming** | K-Means gives numbers, not names: "highway" or "intersection" is a human interpretation. | The automatic naming (`name_cluster`) is declared a **proposal to be confirmed** by looking at the heatmap and map; K-Means renumbering does not break the names because it is based on the values. |
| **Privacy** | GPS traces of real drivers: the data is pseudonymized (`VehId`) but home-work patterns are reconstructable. | Public research dataset used under its terms; no re-identification attempt; in a real product one would need aggregation/anonymization by design (it is the MLOps principle of **governance**: compliance on security, privacy, ethics, transparency). |
| **Geographic generalization** | Everything is learned on **flat Ann Arbor**. The "~0 terrain weight" is a property *of this data*, not of the world. | Limitation declared everywhere (README, report, decision log D1): on real terrain the terrain must be re-measured. |

Method note (from the course): these cases are the **sad paths** of the system. The model that works
on the "average" trip is the happy path; quality is judged by how the system behaves — or
declares it cannot behave — on hybrids, little-used roads, different territories.

## ML lifecycle

The ML lifecycle mirrors the SDLC seen in class (planning → requirements → design →
implementation → test → production → maintenance), with the ML-specific phases. The real
pipeline is in 3 notebooks (`data prep.ipynb` → `consumption prediction.ipynb` + `clustering.ipynb`);
here is each phase with the choices that govern it.

### Data collection
- Consolidation of the 54 VED parquet files, EDA, outlier filters on physical ranges and the "valid
  motion" filter (details in the "Data" section).
- Elevation enrichment: dedup of GPS points to 3 decimals (~111 m) → ~8.7k unique points to
  query, resumable cache (the API is rate-limited). The 30 m grid
  (native SRTM resolution) was explored and **cancelled**: ~2h of calls for a marginal gain on a flat
  city (D10 — an example of a tracked cost/benefit decision).
- Single output: `outputs/ved_enriched.parquet`, the input to everything else.

### Training
- **Consumption**: XGBoost (default + tuned with Optuna), target `maf_per_km` per segment. The decision
  that defines the model is **map-only**: only features known in advance to an ACC are used
  (expected kinematics, geometry, look-ahead on the next segment, context), **excluding**
  RPM/engine load which would be quasi-circular with the target (MAF ≈ RPM × load) (D3).
- **Context/styles**: StandardScaler (mandatory: the features have very different scales, and it is
  demonstrated in the notebook what happens without) → K-Means (k-means++, `n_init=50`) with K chosen via
  Elbow + Silhouette → K=4 for the segments, K=3 for the styles (D9).

### Validation (the "testing" of an ML system)
As seen in class, in ML systems the behavior is not deterministic and depends on the **data**,
not just on the code logic: testing adapts on three levels.
- **Data & component** (transformation tests): sanity checks in NB1 (physical ranges, sample
  trip, post-enrichment statistics) and verification that the elevation cache is complete before the
  merge (the cell *raises* if points are missing: a handled sad path, not ignored).
- **Model functional** (the model *makes sense*, not "exact outputs"): the main proof is not
  R² alone, but the **feature importance** — `stop_fraction` (stop-and-go) dominates, terrain weighs
  ~0.06 → consistent with the D1 reframe, measured and not asserted. For the clustering (without ground truth):
  silhouette + geographic coherence on the Ann Arbor map (the clusters trace real
  highway/arterials/intersections) + readable heatmap profiles.
- **Anti-leakage machinery** (the ML equivalent of "shift-left": the error is prevented upstream,
  not discovered in production):
  - **temporal split per trip** (train on the oldest trips, test on the most recent ones): simulates the
    real use "predict the future";
  - **GroupKFold per vehicle** in the tuning: no vehicle is in both train and validation → the
    model cannot "recognize the vehicle" instead of learning the road (D12);
  - the anticipation features `next_*` are built with `shift(-1)` per (vehicle, trip) and
    carry only the **future of the road**, never the future MAF.
- **Production validation**: not applicable in an exam project; its conceptual equivalent
  is the monitoring below.

### Deploy (conceptual — outside the exam scope)
- **Where**: the consumption model is small (XGBoost on 20 features) → it can run **on-vehicle**;
  the segment classification is precomputable **offline** and distributed as a map layer
  (a context label per segment), refreshed periodically on the backend side.
- **Input contract**: exactly the map-only features; if a feature is not available on board
  (e.g. expected speed profile), the deploy is not possible *by design* — it is the reason the
  engine signals were excluded from training (D3).
- **Environments**: in a real context the release would follow a **DTAP** model (Development →
  Testing → Acceptance → Production): the new model runs in *shadow* alongside the active one
  (same features, predictions compared but not used) before promotion — the A/B/shadow
  deployment of production validation.
- **Versioning** (MLOps principle: data + model + code, not just code): model, feature-schema
  version and context-layer version released **together**; a scikit-learn version mismatch
  between the saving and loading environment was already encountered in the
  project → artifacts are always regenerated in the same environment (trap documented in CLAUDE.md).

### Monitoring (conceptual)
- **Prediction quality**: continuous comparison predicted vs observed (the real `maf_per_km` is
  computable on board after the fact) → rolling MAE/R² per geographic area and per time band.
- **Data drift**: distributions of `speed_mean`, `stop_fraction`, OAT per zone vs the training
  ones; an ACC taken to a hilly city **must** trigger a drift alarm on the
  terrain (we already know the model is blind there).
- **Context health**: fraction of traversed segments that have no label (out of coverage or
  <50 passages) — if it grows, the context layer is aging.

## MLOps

The course's MLOps principles (versioning, reproducibility, automation, monitoring, governance,
collaboration, continuous improvement) applied to this project: **versioning** covers
code *and* state documents (git) and artifacts (regenerable from the notebooks, same environment);
**reproducibility** is guaranteed by fixed seeds (`RANDOM_STATE`), external-data caching and
pipelines with explicit input/output; **monitoring** and **retraining** are designed below;
**collaboration** goes through the documentation discipline described in the work process
(STATE.md at every session, decision log for the choices, conventions in "Further information").

### What to monitor
The three levels described in "Monitoring" above: prediction quality (predicted vs
observed, per zone and time band), drift of the input distributions, health/coverage of the
context layer.

### When retraining would be needed (Continuous Training)
1. **New geographic area** (the sure case): everything is learned on flat Ann Arbor; on a territory
   with real orography the terrain weight must be re-measured — it could go from ~0.06 to dominant.
2. **Seasonal/fleet drift**: OAT, tires, fleet composition change (more hybrids
   → the ICE population on which the model is valid shrinks).
3. **Change of the data source**: new ECU/OBD sampling frequency, new elevation
   source (e.g. switch to a 30 m grid: already explored, D10) → feature recompute + retraining.
4. **Aging of the context layer**: new roads, roadworks, roundabouts replacing intersections
   → periodic re-clustering (it is precomputed offline, hence cheap).

In a CI/CD/CT setup (e.g. GitHub Actions): points 1–3 are *event-based* triggers (new
dataset → retraining + automatic validation workflow), point 4 a periodic *schedule*.

### What signals would indicate a problem
- **Metrics dropping** on the predicted/observed comparison (per zone, not just global: a local drop
  is geographic drift, a global one is model drift).
- **Feature importance changing** after a retraining: if `stop_fraction` stops dominating without
  a domain reason, suspect leakage or a bug in the feature pipeline (it is our "signature").
- **Increase of MAF=0 while moving** on vehicles classified as ICE → powertrain misclassification
  or corrupted data (for hybrids it is instead physiological: ~21–24% of the motion).
- **Silhouette dropping or clusters emptying out** at re-clustering → the chosen K no longer describes
  the road network.
- **Predictions out of physical range** (negative or extreme maf_per_km): trees do not extrapolate →
  input outside the training domain, a typical drift signal.

## Risks, assumptions and limitations

**Assumptions (explicit and verified where possible):**
- MAF ∝ consumption **for ICE only** (verified on the schema: no battery signal in VED).
- The historical passages over a segment describe the segment (holds only with ≥50 passages).
- `Datetime` is constant per trip; intra-trip time is `Timestampms` (verified on the data).

**Risks:**
- **Model shortcuts**: including engine signals would give a higher R² but a model useless
  for ACC use (it learns `MAF ≈ RPM×load`, not the road). Mitigated by construction (map-only, D3).
- **Validation leakage**: mitigated with temporal split + GroupKFold + the rule "never the future target
  in the features".
- **Overfitting to the city**: an accepted and declared risk; the project *demonstrates the method*,
  it does not claim geographic transferability.
- **Technical debt** (Cunningham: "the natural result of writing code about something we do
  not adequately understand"): the project contracted some at the start — the first
  notebooks were built around a hypothesis (orography) not yet validated. The debt was
  **repaid explicitly** with the D1 reframe and the D6–D8/D11 prunings; the residual one is inventoried
  (STATE.md "known issues", documentation gaps declared in this README) instead of hidden.

**Limitations (of the data, not of the method):**
- Almost flat terrain + elevation quantized to ~111 m → slope null in 92% of rows, correlation
  with MAF ~0.004: the role of orography **is not measurable here**. It is the limitation that caused the
  reframe (D1) and it is the first thing to say to whoever evaluates the project.
- Hybrids out of the consumption model (SoC is missing); PHEVs underrepresented (12 vehicles).
- K-Means clustering assumes ~convex clusters in the chosen features; the naming is interpretive.

### Is the project working end-to-end?
**Yes**: the pipeline runs end-to-end (NB1 → `ved_enriched.parquet` → NB2 and NB3) with real results
(R² ~0.76; K=4 with interpretable clusters and a map). A note of transparency on the state at the moment
of submission: after the pruning of June 18 the NB2 must be **re-run in full** so that the outputs
saved in the notebook reflect the latest version of the code (precise state in
`project context/STATE.md` — it is our *Definition of Done*: code and outputs must
match).

**How we could extend it** (extensions already motivated in the context documents):
Lasso as an interpretable baseline alongside XGBoost (in NB2, requires reintroducing scaling);
*hurdle* model for hybrids (blocked: requires SoC data, D4); 30 m elevation grid on
hilly terrain (D10 — makes sense only with real orography); reintroduction of the eco/sport
counterfactual and SEG_LEN sensitivity (removed in D11, results already documented); SHAP on the
consumption model; DP planner "slow down before the climb"; automated tests pytest + CI with GitHub
Actions (first step: extract the pure functions from the notebooks — haversine, segmentation,
look-ahead — into an importable module and test them on the known cases).

**Difficulties encountered** (the main ones, all tracked):
1. The **orographic signal was not there** → project reframe (D1), the biggest and most
   formative difficulty.
2. **Rate-limit of the elevation API** (HTTP 429) → resumable local cache that downloads only the missing
   points.
3. **Hybrids not modelable** without SoC → motivated ICE-only scope (D4).
4. **Environment fragility**: scikit-learn version mismatch on the `.joblib` artifacts and a
   broken `.venv` to recreate → the rule "same environment to save/load" and a disposable venv.
5. **Code↔documents divergence** during fast refactoring moments → STATE.md discipline and
   coherence checks (it already happened, and it already happened to have to fix it).

### What must be clear to whoever joins the team
In reading order: **`STATE.md`** (where we are) → this README + `DECISION_LOG.md` (why we are
here) → `PROJECT_REPORT.md` (the full story) → `CLAUDE.md` (traps not to reintroduce)
→ `discussions.md` sampled. The five takeaways:

1. The project is **not** "the initial idea that succeeded": it is an idea **falsified by the data and reframed**

2. The decision that defines everything is **map-only**: never features not known in advance to an ACC.
3. Consumption is modeled **for ICE only**, for a physical reason (MAF), not out of laziness.
4. The metrics count less than the **explanation**: feature importance and cluster profiles are
   the proof, R² and silhouette the garnish.
5. If something seems arbitrary (250 m, K=4, ~111 m grid), **it is not**: there is an entry in the
   decision log with the discarded alternative and the reason.

### Working conventions (the essentials)
- State documentation **in English**; `STATE.md` is updated at every session that changes
  something — it is the only source of truth on the current state (notebooks can "lie":
  new code with old outputs).
- **Never commit a modified notebook without having re-run it** (or without marking it
  "to re-run" in STATE.md): code and outputs must match.
- No heavy data or artifacts in git: `content/` and `outputs/` stay out (a lesson learned
  in the field: a 79 MB `.joblib` was committed and then removed from history).
- Every scope or methodological decision → an entry in the decision log, **including the no's** (see D10).
- **Definition of Done** of a change: (1) notebook re-run in full in the `.venv`;
  (2) reference numbers confirmed, or the difference explained and documented; (3) STATE.md
  updated and decision tracked; (4) no new reference to removed files; (5) commit with a
  meaningful message.
- **Non-negotiable** ML rules: map-only (D3); anti-leakage (D12: `next_*` only via `shift(-1)`,
  never the future MAF among the inputs, temporal split and GroupKFold are not touched without a tracked
  decision); ICE-only (D4).
- Before long, costly (API) or destructive actions: ask Alex.

| **`maf_per_km`** | The target of the consumption model: MAF integrated over the segment / km traveled = "energy cost per km" of the segment. |
| **ICE / HEV / PHEV** | Combustion / hybrid / plug-in hybrid. The consumption model uses ICE only. |
| **Map-only** | Constraint on the features: only information known **in advance** to an ACC (map, expected speed profile); RPM/load excluded. |
| **Segment** | A ~250 m stretch of a trip: the unit on which consumption is predicted. |
| **Look-ahead / `next_*`** | Features of the **next segment** (future road/speed), built with `shift(-1)`. The "predictive" heart of the ACC. |
| **`stop_fraction`** | Fraction of near-stationary time (speed<2) in the segment: the model's most important feature (stop-and-go). |
| **Trip / `VehId`** | A vehicle's trip / the vehicle's (pseudonymous) identifier. `Datetime` is constant per trip; intra-trip time is `Timestampms`. |
| **Spatial cell** | ~11×8 m bin (lat/lon to 4 decimals) over which passages are aggregated for the segment clustering (kept if ≥50 passages). |
| **Slope / `dz_net`** | Point slope / net elevation change of the segment. In Ann Arbor: a weak signal due to data limitations (elevation quantized ~111 m). |
| **Reframe** | The project's turning point (D1): from "orographic ACC" to "energy + context from the speed profile". |
| **Leakage** | Information from the target (or from the future) that leaks into the features or the validation → inflated metrics and a model unusable in the real use case. |
